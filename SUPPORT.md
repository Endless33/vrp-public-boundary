# Support

## Purpose

This document defines the official support channels for the VRP Public Engineering Boundary.

Different request types are handled through different communication channels.

---

## GitHub Issues

GitHub Issues should be used for:

- documentation corrections
- engineering documentation questions
- public API questions
- OpenAPI questions
- Protocol Buffers questions
- JSON Schema questions
- SDK example issues
- public contract discussions
- compatibility questions
- reproducible defects within this repository

Before opening an issue:

- verify that the question concerns a public artifact;
- search existing issues;
- remove confidential information;
- include a minimal reproducible example whenever possible.

---

## Not Supported Through GitHub Issues

The following requests will not be answered through public issues:

- Protected Runtime source code
- proprietary algorithms
- internal runtime implementation
- authority engine implementation
- recovery engine implementation
- replay protection implementation
- runtime mathematics
- customer deployments
- commercial runtime packages
- reverse-engineering assistance

---

## Security Reports

Potential security issues concerning public repository artifacts should be reported according to:

```text
SECURITY.md
```

Please do not publish active security vulnerabilities before responsible disclosure.

---

## Technical Evaluation

Organizations interested in technical evaluation should review:

```text
docs/TECHNICAL_EVALUATION.md
docs/PILOT_PROGRAM.md
docs/PILOT_WORKFLOW.md
```

Technical evaluation focuses on:

- engineering compatibility
- integration feasibility
- validation methodology
- evidence verification
- deployment planning

## Support Policy

Public support is provided on a best-effort basis.

Commercial support, deployment assistance, implementation guidance, and runtime access are available only through approved Pilot engagements or separate commercial agreements.

---

## Engineering Principle

Public engineering interfaces are open.

Protected implementation remains proprietary.

Independent validation remains reproducible.
