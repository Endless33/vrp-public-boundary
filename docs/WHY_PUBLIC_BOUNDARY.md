# Why VRP Public Engineering Boundary Exists

## Overview

VRP Public Engineering Boundary exists to provide a stable and transparent engineering interface without exposing the protected runtime implementation.

The objective is straightforward:

Allow engineers to understand, integrate, validate, and evaluate VRP while preserving the proprietary runtime responsible for protocol execution.

This repository is intentionally public.

The runtime is intentionally protected.

---

# Why Not Open The Entire Runtime?

VRP is built around years of research, engineering, validation, and runtime development.

The protected runtime contains proprietary implementation details that are not required for external engineering evaluation.

Publishing every implementation detail would not improve integration.

It would simply expose proprietary technology.

Instead, VRP separates the engineering surface from the execution engine.

---

# Public Engineering Boundary

The public boundary is designed to expose everything required for professional engineering collaboration.

Examples include:

- Engineering documentation
- Public SDK interfaces
- Integration contracts
- Validation specifications
- Evidence formats
- Example implementations
- Public APIs
- Pilot documentation

This allows organizations to understand how VRP integrates into existing environments without requiring access to protected runtime internals.

---

# Protected Runtime

The protected runtime remains private.

This includes, but is not limited to:

- Runtime decision engine
- Session continuity implementation
- Authority algorithms
- Recovery engine
- Internal protocol mathematics
- Proprietary runtime architecture
- Commercial deployment components

These components are available only through authorized commercial engagements.

---

# Engineering Philosophy

VRP is designed around a simple principle.

Transparency where transparency creates interoperability.

Protection where protection preserves engineering value.

The engineering interface should be open.

The implementation does not have to be.

---
